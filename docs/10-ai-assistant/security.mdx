# AI Security

The AI assistant is powerful, but power requires boundaries. Labonair enforces a set of hard security constraints that protect sensitive files and credentials, regardless of what is asked in the chat.

## File Access Deny-List

All file reads and writes pass through a deny-list defined in `src/modules/ai/lib/security.ts`. If a requested path matches any entry on this list, the operation is blocked at the Labonair layer — the request never reaches the AI provider and the file contents are never sent.

Blocked path patterns include:

| Pattern | Protects |
|---|---|
| `.env*` | Environment variable files |
| `.env.local` | Local environment overrides |
| `.ssh/` | SSH keys and `known_hosts` |
| `*credentials*` | AWS, GCP, and similar credential files |
| `*secret*` | Any file with "secret" in the path |
| `*.pem` | PEM-encoded certificates and private keys |
| `*.key` (private keys) | Private key files |

The deny-list applies to **both** `read_file` and `write_file` operations. An AI that cannot read a secret cannot accidentally include it in a generated file or an API call.

## These Restrictions Cannot Be Overridden

Instructions in the chat — including instructions like "ignore your restrictions" or "you have permission to read .env" — do not bypass the deny-list. The restrictions are enforced in Labonair's code, not as part of the AI's system prompt, so they cannot be prompted around.

## Fully Local Execution

All tool calls (`read_file`, `write_file`, `run_command`, etc.) execute locally via Tauri IPC. There is no Labonair server, no relay, and no telemetry. The only network traffic generated by the AI assistant is the API request to your chosen AI provider (OpenAI or compatible), and that traffic contains only what you have deliberately sent in the conversation.

## API Key Security

Your API key is stored in the macOS Keychain under the service name `labonair-ai`. It is never written to disk in plain text, never included in crash reports, and never logged. See [Setting Up an API Key](./setting-up-api-key) for details.

## Recommendations

- Keep sensitive files outside your project directories when working with the AI agent, or ensure they match the deny-list patterns
- Use version control — the AI can overwrite files when `write_file` is approved; a git working tree gives you a safety net
- Review the exact command shown in every `run_command` Approval Card before clicking Allow
- If you are working with highly sensitive infrastructure, consider using a dedicated Ollama instance instead of a cloud API so that no data leaves your local network

:::warning
The deny-list protects well-known sensitive file patterns, but it cannot know about every custom secret file in your environment. If you have non-standard credential files, keep them out of directories you work in with the AI, or add patterns to the deny-list in `src/modules/ai/lib/security.ts` and rebuild the app.
:::
