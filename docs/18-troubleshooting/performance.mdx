# Performance Issues

Labonair is designed to be lightweight, but certain usage patterns can cause performance degradation. This page covers the most common scenarios and how to address them.

## SFTP Directory Listing Is Slow

Browsing a directory with hundreds or thousands of files takes several seconds.

**Cause:** The slowness is in the network round-trip to fetch the directory listing over SFTP, not in Labonair's rendering. The file list is virtualized (only visible rows are rendered), so a 10,000-file directory renders as fast as a 10-file directory once the data arrives.

**Resolution:** There is no client-side fix for slow SFTP listing over a high-latency connection. Narrow your working directory to subdirectories with fewer files when possible.

## Terminal Lag or Input Delay

Keystrokes feel delayed or the terminal output lags behind input.

**Cause:** A runaway process generating massive output is flooding the PTY buffer. Common examples: `find /` without filters, `tail -f` on a very active log file, or a loop with no delay.

**Resolution:** Press `Ctrl+C` to interrupt the foreground process. If the terminal is unresponsive, close the tab — the PTY process is cleaned up automatically by the Rust backend. For SSH sessions, the session is terminated cleanly.

## High CPU Usage

Labonair's process is consuming significant CPU even when idle.

**Cause:** The WebGL renderer in xterm.js uses GPU/CPU resources proportional to terminal activity. A tab with continuous fast-scrolling output (e.g., a build log) can sustain elevated CPU usage.

**Resolution:**
- Disable the WebGL addon in Settings → Terminal to use the canvas 2D renderer (lower CPU overhead on older Intel Macs)
- Close tabs with high-output processes when you no longer need them
- On Apple Silicon, CPU impact is typically minimal due to efficient GPU scheduling

## Memory Usage Grows Over Time

Labonair's memory footprint increases over a long session.

**Cause:** Each terminal tab maintains an xterm.js scrollback buffer in the WebView's JavaScript heap. A large scrollback limit across many tabs accumulates significant memory.

**Resolution:**
- Reduce the scrollback buffer size in Settings → Terminal (default is 5,000 lines; try 1,000–2,000 for long sessions)
- Close unused terminal and SFTP tabs
- The `Clear Terminal` command (`Cmd+K` when terminal is focused) clears the xterm.js buffer and frees the associated memory immediately

## App Feels Sluggish After a Long Session

General UI responsiveness degrades after many hours of continuous use.

**Cause:** Tauri's WKWebView can accumulate garbage collected JavaScript heap memory over very long sessions with heavy AI usage, many tab switches, or frequent theme changes.

**Resolution:** Restart Labonair. Running terminal sessions in the Rust backend are terminated, but SSH sessions can be reconnected quickly via the Host Quick-Connect in the Command Palette.

:::tip
For production monitoring use cases where Labonair runs continuously for days, consider closing and reopening the app once per day to reset the WebView memory state.
:::
